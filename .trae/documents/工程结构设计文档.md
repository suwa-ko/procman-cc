## 1. 设计目标与原则

### 1.1 设计目标

本项目核心定位为**单人开发 + AI 协作的极简采购管理系统**。建立 monorepo 的核心目的不是打造企业级通用基建，而是**强制约束依赖方向，避免 AI 写出依赖反转、跨层引用、环境污染等错误代码**。所有结构设计围绕以下四大原则：

- **精简**：包数量与基建能力只保留最低必要集合，拒绝过度工程化。
- **可控**：依赖方向单一且可机器校验，AI 改动一旦越界立即报错。
- **防 AI 出错**：通过 ESLint 规则、循环依赖检测、环境变量约束等工具链，将架构约束固化为可执行检查。
- **快速落地**：分阶段实施，每阶段可独立验证交付，对齐 PRD 一期需求边界，不做超前预留。

### 1.2 设计原则

| 原则             | 说明                                                                 |
|------------------|----------------------------------------------------------------------|
| 分层依赖         | 基础层 → 数据/服务层 → 业务层，严格单向依赖，下层绝不反向引用上层     |
| 接口导向         | package 间通过类型与接口交互，不直接依赖具体实现                      |
| 依赖注入         | 包之间通过上层注入连接，而非硬编码引用（如 log 级别由 apps 注入）     |
| 零内部依赖       | 最底层包（`@ps/types-base`、`@ps/utils`、`@ps/env-config`）零内部依赖，保证可独立使用与测试 |
| 命名空间统一     | 所有内部 package 统一使用 `@ps/` 作用域前缀                          |
| 零业务耦合       | `packages/` 下的包不感知具体业务逻辑，只提供通用能力                 |
| 扩展接口预留     | 所有扩展能力仅预留接口，一期不落地实现，严格对齐 PRD 一期边界         |

## 2. 技术栈选型

| 层面         | 选型                              | 说明                                           |
|--------------|-----------------------------------|------------------------------------------------|
| 语言         | TypeScript                        | 全栈统一类型系统                               |

| 依赖校验     | eslint-plugin-import + dpdm       | 禁止下层引用上层 + 循环依赖检测                |

## 4. pnpm workspace 配置

### 4.1 pnpm-workspace.yaml

```yaml
packages:
  - "packages/*"
  - "apps/*"
```

### 4.2 根 package.json（关键字段）

```jsonc
{
  "name": "purchase-system",
  "private": true,
  "scripts": {
    "dev": "pnpm -r --parallel run dev",
    "build": "pnpm -r run build",
    "lint": "pnpm -r run lint",
    "test": "pnpm -r run test",
    "typecheck": "pnpm -r run typecheck",
    "check:circular": "dpdm --exit-code circular packages/*/src apps/*/src"
  },
  "devDependencies": {
    "typescript": "^5.4.0",
    "vite": "^5.4.0",
    "vite-plugin-dts": "^4.0.0",
    "eslint": "^8.57.0",
    "eslint-plugin-import": "^2.29.0",
    "dpdm": "^3.14.0",
    "prettier": "^3.2.0",
    "vitest": "^1.6.0"
  }
}
```

> **说明**：已移除 `tsup`，所有库统一使用 `vite` + `vite-plugin-dts` 构建。

### 4.3 库模式 Vite 构建配置（各包共享模板）

各 `packages/*` 包的 `vite.config.ts` 遵循统一模板：

```ts
import { defineConfig } from 'vite'
import dts from 'vite-plugin-dts'

export default defineConfig({
  plugins: [dts({ rollupTypes: true })],
  build: {
    lib: {
      entry: 'src/index.ts',
      formats: ['es'],
      fileName: () => 'index.mjs',
    },
  },
})
```

### 4.4 内部包引用方式

内部包通过 `workspace:` 协议引用，确保本地联动：

```jsonc
// apps/web/package.json
{
  "dependencies": {
    "@ps/web-kit": "workspace:*",
    "@ps/mock": "workspace:*",
    "@ps/env-config": "workspace:*",
    "@ps/log": "workspace:*",
    "@ps/contracts": "workspace:*",
```

## 5. 共享基础包（packages）

基础包共 9 个，按依赖方向自下而上分四层排列：

```
第一层（零内部依赖）：@ps/types-base  @ps/utils  @ps/env-config  @ps/log
第二层（依赖第一层）：@ps/contracts  @ps/mock
第三层（依赖第一二层）：@ps/db  @ps/pdf
第四层（依赖第一二三层）：@ps/web-kit
```

### 5.1 @ps/types-base — 基础类型包

**职责**：存放与业务无关的基础类型定义，是整个项目类型系统的根基。

**运行环境**：universal（纯类型，不依赖任何运行时 API）。

**目录结构**：

```
packages/types-base/
├── package.json
├── tsconfig.json
├── vite.config.ts
├── src/
│   ├── index.ts
│   ├── pagination.ts       # 分页类型：PaginationParams, PaginatedResponse<T>
│   ├── api-response.ts     # 通用响应类型：ApiResponse<T>, ApiError
│   ├── id.ts               # ID 类型定义
│   └── common.ts           # 通用类型：Nullable<T>, DeepPartial<T> 等
└── tests/
```

**核心能力**：

- `PaginationParams`：分页请求参数（page, pageSize）。
- `PaginatedResponse<T>`：分页响应结构（data, total, page, pageSize）。
- `ApiResponse<T>`：统一 API 响应包装（code, data, message）。
- `ApiError`：统一错误结构。
- 通用泛型工具类型。

**依赖**：零内部依赖（最底层，禁止引用任何其他 `@ps/` 包）。

---

### 5.2 @ps/utils — 纯函数工具包

**职责**：存放纯函数工具（如编码生成、金额计算、日期格式化），可被任何包直接引用。

**运行环境**：universal（纯函数，不依赖任何运行时 API，不依赖任何内部包）。

**目录结构**：

```
packages/utils/
├── package.json
├── tsconfig.json
├── vite.config.ts
├── src/
│   ├── index.ts
│   ├── code-generator.ts   # 编码生成器（前缀-年份-流水号）
│   ├── money.ts            # 金额计算（round 到两位、加法、乘法）
│   ├── date.ts             # 日期格式化
│   └── string.ts           # 字符串工具（脱敏、截断等）
└── tests/
```

**核心能力**：

- 编码生成规则：`generateCode(prefix, year, seq)` → `SUP-2026-0001` 格式。
- 金额计算：`roundTo2(n)`、`multiplyPrice(price, qty)`。
- 日期格式化：`formatDate(date, pattern)`。
- 字符串脱敏：`maskSensitive(str)`。

**依赖**：零内部依赖（不依赖任何其他 `@ps/` 包）。

### 5.4 @ps/env-config — 环境配置包

**职责**：负责加载环境配置（开发/生产预设、校验），是项目运行环境的唯一入口。

**运行环境**：universal（配置加载逻辑均可在浏览器与 Node.js 运行）。

**目录结构**：

```
packages/env-config/
├── package.json
├── tsconfig.json
├── vite.config.ts
├── src/
│   ├── index.ts
│   ├── types.ts              # EnvConfig 接口定义
│   ├── loader.ts             # 环境变量加载逻辑
│   ├── presets/
│   │   ├── development.ts    # 开发环境：Supabase 本地实例
│   │   └── production.ts    # 生产环境：Supabase 线上
│   └── validator.ts          # 必填项校验
└── tests/
```

**核心能力**：

- 统一 `EnvConfig` 接口，包含：`supabaseUrl`、`supabaseAnonKey`、`supabaseServiceKey`、`apiBaseUrl`、`appName`、`logoUrl`、`nodeEnv`、`logLevel` 等字段。
- **仅保留「开发 / 生产」两套环境预设**。
- 根据 `NODE_ENV` 自动加载对应 preset，支持环境变量覆盖。
- 启动时校验必填配置项，缺失时使用 `console.error` 直接输出错误并退出，不调用任何日志库。

**依赖注入说明**：

- `@ps/env-config` 零内部依赖，不引用 `@ps/log`。
- 上层应用（`apps/web`、`apps/api`）在启动时读取 `@ps/env-config` 的配置后，再初始化 `@ps/log` 并传入日志级别（`config.logLevel`）。
- `@ps/env-config` 与 `@ps/log` 之间**无直接引用**，通过 apps 层依赖注入连接。

**依赖**：零内部依赖（最底层，不依赖任何其他 `@ps/` 包）。

### 5.5 @ps/log — 统一日志包

**职责**：提供统一的日志接口，支持四级日志与控制台输出。

**运行环境**：universal（日志逻辑均可在浏览器与 Node.js 运行）。

**目录结构**：

```
packages/log/
├── package.json
├── tsconfig.json
├── vite.config.ts
├── src/
│   ├── index.ts
│   ├── types.ts              # LogLevel, Logger 接口, LoggerConfig
│   ├── logger.ts             # Logger 核心实现
│   └── transports/
│       └── console.ts        # 控制台输出（一期唯一实现）
└── tests/
```

**核心能力**：

- 统一 `debug / info / warn / error` 四级日志。
- **仅保留控制台输出通道**（`console.ts`），移除 HTTP 上报、多格式化器、多 transport 抽象等企业级特性。
- 内置敏感字段脱敏（密码、Token 等自动打码），满足安全要求。
- **Logger 接口预留 `addTransport` 扩展点，但一期不落地任何额外实现**。

**依赖注入说明**：

- `@ps/log` 零内部依赖，不引用 `@ps/env-config`。
- Logger 通过 `createLogger(config: LoggerConfig)` 工厂函数创建，`config` 中包含 `level` 字段。
- 上层应用启动时：
  ```ts
  // apps/api/src/index.ts（示例）
  import { loadConfig } from '@ps/env-config'
  import { createLogger } from '@ps/log'

  const config = loadConfig()           // 读取环境配置
  const logger = createLogger({         // 注入日志级别
    level: config.logLevel,
  })
  ```
- `@ps/log` 内部绝不直接引用 `@ps/env-config`，消除初始化死循环风险，且可独立测试。

**依赖**：零内部依赖（不依赖任何其他 `@ps/` 包）。

### 5.6 @ps/db — Supabase 数据访问层

**职责**：封装对 Supabase 的读写操作，向上层提供领域化的数据访问函数。**仅做原子数据 CRUD 与查询封装，不包含任何业务规则与判断**。

**运行环境**：node（服务端专用，使用 `service_role` key）。

**拆包决策说明**：

曾考虑将 `@ps/db` 拆分为 `@ps/db-core`（Client 初始化 + 基础 CRUD 抽象）与 `@ps/db-repositories`（业务 Repository 实现）两个包。经评估，**决定暂不拆包**，理由如下：

1. **精简优先**：当前项目为单人开发，Repository 数量有限（7 个），拆包引入的管理成本大于收益。
2. **AI 出错风险**：拆包后 AI 更容易在 db-core 与 db-repositories 之间混淆引用方向。
3. **可内聚替代**：通过目录结构隔离即可达到同样的关注点分离效果，无需物理拆包。

**后续扩展预留**：当 Repository 数量超过 15 个或需要独立发布 db-core 供其他项目使用时，再拆分为 `@ps/db-core` + `@ps/db-repositories`。当前在 `@ps/db` 内部通过目录结构明确隔离。

**目录结构**：

```
packages/db/
├── package.json
├── tsconfig.json
├── vite.config.ts
├── src/
│   ├── index.ts              # 导出 client 与各 repository
│   ├── core/                 # DB 机制层（未来拆包时的 @ps/db-core）
│   │   ├── client.ts         # Supabase Client 创建
│   │   ├── types.ts          # 数据库行类型映射
│   │   └── base-repository.ts # 基础 CRUD 抽象类
│   ├── repositories/         # 业务 Repository 层（未来拆包时的 @ps/db-repositories）
│   │   ├── supplier.repo.ts
```

### 5.7 @ps/pdf — PDF 生成库

**职责**：基于 HTML 模板引擎渲染合同数据并生成 PDF 文件。**仅负责模板渲染与 PDF 生成，绝不直接访问数据库**，保持职责单一。

**运行环境**：node（服务端专用，依赖 Puppeteer）。

**目录结构**：

```
packages/pdf/
├── package.json
├── tsconfig.json
├── vite.config.ts
├── src/
│   ├── index.ts              # 导出 renderPdf 函数
│   ├── renderer.ts           # HTML → PDF 核心转换（Puppeteer）
│   ├── template-engine.ts    # Handlebars 模板渲染
│   ├── templates/            # 内置默认模板（兜底用）
│   │   ├── nda.default.html
│   │   └── purchase.default.html
│   ├── assets/               # PDF 资源
│   │   ├── fonts/            # 中文字体（宋体/黑体）
│   │   └── logo/             # 企业 Logo 占位
│   └── types.ts              # PdfRenderContext 类型
└── tests/
```

**核心能力**：

- 接收 `PdfRenderContext`（模板字符串 + 合同数据上下文），返回 PDF Buffer 或 HTML 流。
- 使用 Handlebars 渲染模板 HTML。
- 通过 Puppeteer 将 HTML 转为 PDF，注入中文字体、Logo、水印样式。
- 支持在线预览（返回 HTML 流）和文件下载（返回 PDF Buffer）两种模式。
- 文件命名格式：`合同编号-合同名称.pdf`。

**模板与 PDF 联动链路（明确职责边界）**：

```
1. apps/api 接收 PDF 生成请求（合同 ID）
2. apps/api 通过 @ps/db 查询合同完整数据 + 对应模板的 HTML 内容
3. apps/api 组装 PdfRenderContext（模板字符串 + 合同数据上下文）
4. apps/api 调用 @ps/pdf 的 renderPdf 函数
5. @ps/pdf 仅做渲染与转换，返回 PDF Buffer
6. apps/api 返回给前端
```

> `@ps/pdf` 绝不直接调用 `@ps/db`，数据由调用方（`apps/api`）注入。

**依赖**：`@ps/log`（日志）、`@ps/contracts`（合同类型定义）、`@ps/types-base`。不依赖 `@ps/db`。

**核心能力 — React Hooks**：

`useQuery` 和 `useMutation` 是前端请求状态管理的核心 Hook，底层调用 `@ps/web-kit/request` 的 fetch 客户端，并将响应数据自动映射为 `@ps/contracts` 中定义的契约类型。

**`useQuery<T>`** — GET 请求 Hook：

```ts
// 设计思路：声明式获取数据，自动管理 loading / error / data 状态
interface UseQueryResult<T> {
  data: T | null
  loading: boolean
  error: Error | null
  refetch: () => void
}

function useQuery<T>(
  url: string,
  options?: { params?: Record<string, string>; enabled?: boolean }
): UseQueryResult<T>

// 使用示例：
const { data: suppliers, loading, error } = useQuery<SupplierListDTO>('/api/suppliers', {
  params: { status: 'active' },
  enabled: true,  // 条件请求，默认 true
})
```

**`useMutation<TData, TVariables>`** — POST/PUT/DELETE 请求 Hook：

```ts
// 设计思路：命令式触发请求，管理提交状态，支持 onSuccess 回调
interface UseMutationResult<TData, TVariables> {
  data: TData | null
  loading: boolean
  error: Error | null
  mutate: (variables: TVariables) => Promise<TData>
}

function useMutation<TData, TVariables>(
  url: string,
  options?: { method?: 'POST' | 'PUT' | 'DELETE'; onSuccess?: (data: TData) => void }
): UseMutationResult<TData, TVariables>

// 使用示例：
const { mutate: createContract, loading } = useMutation<ContractDTO, CreateContractDTO>(
  '/api/contracts',
  {
    method: 'POST',
    onSuccess: (data) => {
      toast.success(`合同 ${data.code} 创建成功`)
      navigate(`/contracts/${data.id}`)
    },
  }
)
```

**核心能力 — UI 组件库**：

- 基础原子组件封装原生 HTML 元素，统一样式与交互。
- 业务复合组件组合原子组件，内置表单校验逻辑（引用 `@ps/contracts` 的 Zod schema）。
- 状态标签组件统一渲染各实体的状态枚举。
- 二次确认对话框，用于合同作废、定价失效、供应商淘汰等不可逆操作。
- 组件接收数据而非自行请求（关注点分离）。

**依赖**：`@ps/log`（日志）、`@ps/env-config`（baseURL 注入）、`@ps/contracts`（DTO 类型与校验）、`@ps/types-base`。**禁止依赖 `@ps/db`、`@ps/pdf` 等 Node 专属包**。

## 9. 开发与构建工作流

### 9.1 本地开发

```bash
# 安装所有依赖（workspace 联动）
pnpm install

# 并行启动所有 package 的 dev 模式（watch）
pnpm dev

# 仅启动前端 + 后端
pnpm --filter @ps/web dev
pnpm --filter @ps/api dev
```

### 9.2 构建与校验

```bash
# 全量构建（按依赖拓扑顺序）
pnpm build

# 全量类型检查
pnpm typecheck

# 全量 lint（含依赖方向校验）
pnpm lint

# 循环依赖检测（必跑）
pnpm check:circular

# 全量测试
pnpm test
```

### 9.3 单包操作

```bash
# 仅构建某个包
pnpm --filter @ps/web-kit build

# 仅测试某个包
pnpm --filter @ps/db test

# 给某个包添加外部依赖
pnpm --filter @ps/pdf add puppeteer
```

### 9.4 环境变量管理

- 根目录 `.env.example` 列出所有需要的环境变量。
- 各 app 在本地创建 `.env.local`（不提交），由 `@ps/env-config` 加载。
- 关键变量：

| 变量名                  | 说明                          | 开发默认值                  |
|-------------------------|-------------------------------|-----------------------------|
| `NODE_ENV`              | 运行环境                      | development                 |
| `SUPABASE_URL`          | Supabase 实例地址             | `http://localhost:54321`    |
| `SUPABASE_ANON_KEY`     | Supabase 匿名 Key             | 本地实例 anon key           |
| `SUPABASE_SERVICE_KEY`  | Supabase 服务 Key（仅后端）   | 本地实例 service key        |
| `API_BASE_URL`          | 后端 API 地址（前端用）       | `http://localhost:3000`     |
| `APP_NAME`              | 企业名称                      | 采购管理系统                |
| `LOGO_URL`              | 企业 Logo 地址                | 本地占位资源                |
| `LOG_LEVEL`             | 日志级别                      | debug（开发）/ info（生产） |

### 9.5 AI 协作开发规范

本节为 AI 协作开发的三条铁则，违反则必须立即停止并修正：

**铁则一：单包聚焦**

每次开发仅聚焦一个 package，不跨包并行修改。如需修改多个包，按依赖顺序逐个完成。

**铁则二：自下而上验证**

底层包必须通过「构建 + 类型检查 + 测试 + lint + 循环依赖检测」全量验证后，再开发上层包。禁止在底层包未验证通过时就开始上层开发。

**铁则三：影响范围声明**

修改基础包（`@ps/types-base`、`@ps/utils`、`@ps/env-config`、`@ps/log`）必须说明影响范围，并验证所有上层依赖包的兼容性。修改 `@ps/contracts` 的 DTO 类型时，必须同步检查 `@ps/db`、`@ps/pdf`、`@ps/mock`、`@ps/web-kit`、`apps/api`、`apps/web` 的类型一致性。

### 9.6 AI_GUIDELINES.md 说明

根目录新增 `AI_GUIDELINES.md` 规则文件，作为 AI 编码前置参考，内容包括：

- **命名规范**：package 名称、文件命名、变量命名约定
- **导入顺序**：外部依赖 → 内部包 → 相对路径，分组排列
- **架构约束**：依赖方向、环境隔离、职责边界（详见本文档第 7 章）
- **依赖注入约定**：`@ps/env-config` 与 `@ps/log` 不直接引用，通过 apps 层注入
- **单包开发原则**：聚焦单包、自下而上验证、影响范围声明（详见 9.5 节）

AI 在开始任何编码任务前，必须先阅读 `AI_GUIDELINES.md` 与本文档。

## 10. 分阶段实施路径

严格按依赖顺序分 4 个阶段落地，每阶段完成后必须通过单包交付验收标准（见第 11 章）方可进入下一阶段。

### 阶段一：工程底座搭建

**目标**：初始化 pnpm workspace、根目录统一配置，验证工具链可用。

**任务**：

- 创建 `pnpm-workspace.yaml`、根 `package.json`、`tsconfig.base.json`
- 配置根 ESLint（含 `eslint-plugin-import` 依赖方向规则）、Prettier
- 配置 `dpdm` 循环依赖检测脚本
- 验证 Vite + `vite-plugin-dts` 库模式构建可用
- 创建 `AI_GUIDELINES.md`

**验收**：工具链全部可用，空项目能通过 lint 与循环依赖检测。

### 阶段二：底层基础包

**目标**：实现零内部依赖的 4 个基础包 + 契约层 2 个包。

**任务**：

- `@ps/types-base`：定义基础泛型类型
- `@ps/utils`：实现编码生成、金额计算、日期格式化、字符串脱敏
- `@ps/env-config`：实现环境配置加载（开发/生产预设、校验）
- `@ps/log`：实现日志核心（四级 + 控制台 + 敏感字段脱敏）
- `@ps/contracts`：定义所有业务枚举、DTO、Zod schema、常量
- `@ps/mock`：实现各业务模块的 Faker 数据工厂 + Fixtures
- 全量验证：build、typecheck、test、lint、check:circular

**验收**：6 个包均通过单包交付四要素。

### 阶段三：能力层包

**目标**：实现 `@ps/db`、`@ps/pdf`、`@ps/web-kit`，逐个包验证交付。

**任务**：

- `@ps/db`：封装 Supabase Client、基础 CRUD 抽象类、各领域 Repository、复杂查询
- `@ps/pdf`：实现 Handlebars 模板渲染、Puppeteer HTML→PDF 转换、内置默认模板
- `@ps/web-kit`：实现 fetch 封装 + 拦截器 + `useQuery`/`useMutation` Hooks + UI 组件库
- 每个包独立验证：build、typecheck、test、lint、check:circular

**验收**：3 个包均通过单包交付四要素，且不违反依赖方向约束。

### 阶段四：业务应用层

**目标**：实现 `apps/api`、`apps/web`，按业务模块联调。

**任务**：

- `apps/api`：实现 Hono 服务、依赖注入启动、中间件、路由、service 层业务规则编排
- `apps/web`：实现依赖注入启动、页面路由、状态管理、MSW 拦截层、页面组装
- 按业务模块（供应商→物料→定价→模板→合同→PDF）逐个联调

**验收**：前后端联调通过，核心业务流程跑通，全量验证通过。

## 11. 单包交付验收标准

每个 package 开发完成后，必须通过以下「单包交付四要素」强制验收，缺一不可：

| 验收项       | 命令                            | 要求                                   |
|--------------|---------------------------------|----------------------------------------|
| 构建通过     | `pnpm --filter <pkg> build`     | 无错误，Vite 产物 + `.d.ts` 正常生成  |
| 类型安全     | `pnpm --filter <pkg> typecheck` | 严格模式下类型检查通过，无隐式 any     |
| 测试通过     | `pnpm --filter <pkg> test`      | 核心功能单元测试覆盖，全量用例通过     |
| 代码规范     | `pnpm --filter <pkg> lint`      | ESLint 无错误、无警告                   |

**补充要求**：

- 所有包必须通过 `pnpm check:circular` 循环依赖检测。
- 修改基础包（零内部依赖的 4 个包 + `@ps/contracts`）时，必须额外运行全量 `pnpm typecheck` 验证上层包兼容性。
- 验收未通过的包不得进入下一阶段开发。
