## 1. 产品概述

### 1.1 项目背景

为一家计算机零部件制造企业构建采购管理系统，覆盖供应商管理、物料品类管理、定价管理、合同模板管理、合同管理及 PDF 导出等核心采购链路。系统定位为聚焦核心能力的轻量级应用，适合单人开发与快速迭代。

### 1.2 目标用户

| 角色   | 职责描述                                                     |
|--------|--------------------------------------------------------------|
| 采购员 | 录入供应商信息、维护物料与定价数据、创建与管理合同、导出 PDF |

> **说明**：当前版本仅设单一角色。系统预留扩展其他角色（如审批人、法务等）的架构空间，具体实现不属于本期范围。

### 1.3 产品边界

| 维度         | 范围                                                                 |
|--------------|----------------------------------------------------------------------|
| **包含**     | 供应商管理、物料与品类管理、人员信息展示与账号认证、定价管理、合同模板管理、合同管理、PDF 导出 |
| **不包含**   | 审批工作流、多角色权限体系、采购订单/入库/付款等下游执行、财务系统对接、招投标管理、库存管理 |

### 1.4 系统编码规则

系统中所有业务实体编码统一采用 `前缀-年份-4位流水号` 格式，保证全局唯一且人工可读：

| 实体类型     | 前缀 | 示例              |
|--------------|------|-------------------|
| 供应商       | SUP  | SUP-2026-0001     |
| 物料         | MAT  | MAT-2026-0001     |
| 定价         | PRC  | PRC-2026-0001     |
| 合同         | CTT  | CTT-2026-0001     |
| 合同模板     | TPL  | TPL-2026-0001     |

**规则细节**：
- 流水号为 4 位数字，按年份独立计数，每年从 0001 起始。
- 编码在实体创建时由系统自动生成，用户不可编辑。

## 2. 功能需求详述

### 2.1 供应商管理

#### 2.1.1 功能列表

| 编号    | 功能名称       | 功能描述                                                       |
|---------|----------------|----------------------------------------------------------------|
| SUP-001 | 供应商信息录入 | 录入供应商的名称、统一社会信用代码、联系人、联系电话、邮箱、注册地址、经营范围等信息 |
| SUP-002 | 供应商信息编辑 | 修改已有供应商的字段信息                                       |
| SUP-003 | 供应商列表查询 | 支持按供应商名称、编码、状态等条件进行模糊搜索与精确筛选       |
| SUP-004 | 供应商状态管理 | 管理供应商生命周期状态：已准入 → 冻结 → 淘汰                   |
| SUP-005 | 供应商详情查看 | 查看供应商完整信息及其关联的物料、定价、合同记录               |

#### 2.1.2 数据字段定义

| 字段名             | 类型   | 必填 | 说明                                               |
|--------------------|--------|------|----------------------------------------------------|
| 供应商编码          | 字符串 | 是   | 系统自动生成，遵循 `SUP-YYYY-NNNN` 编码规则，全局唯一 |
| 供应商名称          | 字符串 | 是   | 企业工商注册全称                                     |
| 统一社会信用代码    | 字符串 | 是   | 18 位，用于企业唯一性标识                             |
| 联系人             | 字符串 | 否   |                                                    |
| 联系电话           | 字符串 | 否   |                                                    |
| 联系邮箱           | 字符串 | 否   |                                                    |
| 注册地址           | 字符串 | 否   |                                                    |
| 经营范围           | 文本   | 否   |                                                    |
| 状态               | 枚举   | 是   | 可选值：已准入 / 冻结 / 淘汰                          |
| 备注               | 文本   | 否   |                                                    |

#### 2.1.3 业务规则

- **状态流转**：新录入供应商默认状态为"已准入"；"已准入"可流转至"冻结"或"淘汰"；"冻结"可恢复至"已准入"或转为"淘汰"；"淘汰"为终态，不可逆转。
- **级联约束**：供应商变为"冻结"或"淘汰"后，不可再为其创建新的有效定价或合同；已有合同不受影响。
- **删除限制**：供应商已被定价或合同引用时，不允许物理删除。

| MAT-005 | 物料详情查看 | 查看物料完整信息及其关联的供应商定价记录                         |

|----------|--------|------|----------------------------------------------------|

| 品类名称 | 字符串 | 是   |                                                    |
| 父级品类 | 引用   | 否   | 指向父级品类，为空表示根节点                         |

#### 2.2.3 物料数据字段定义

| 字段名   | 类型   | 必填 | 说明                                 |
|----------|--------|------|--------------------------------------|
| 物料编码 | 字符串 | 是   | 系统生成，遵循 `MAT-YYYY-NNNN` 编码规则 |
| 物料名称 | 字符串 | 是   |                                      |
| 规格型号 | 字符串 | 否   |                                      |

| 所属品类 | 引用   | 是   | 每个物料必须归属一个品类               |
| 描述     | 文本   | 否   |                                      |

#### 2.2.4 业务规则

- **品类删除限制**：品类下存在子品类或关联物料时，不允许删除。
- **物料停用约束**：物料停用后不可再创建新的有效定价。
- **物料删除限制**：物料已被定价引用时，不允许物理删除。

### 2.3 人员与账号管理

> **设计前提**：部门与人员主数据由外部系统（如企业 HR 系统）维护，本系统仅存储人员唯一标识和姓名用于合同关联展示，不维护完整的人员主数据。

#### 2.3.1 功能列表

| 编号    | 功能名称         | 功能描述                                                         |
|---------|------------------|------------------------------------------------------------------|
| ORG-001 | 人员信息引用     | 通过人员唯一标识关联外部人员主数据，仅存储标识和姓名用于合同展示   |
| ORG-002 | 账号注册与认证   | 提供采购员账号的注册和登录认证功能                               |
| ORG-003 | 经办人默认填充   | 登录后自动将当前用户关联的人员姓名填入合同的经办人字段             |

#### 2.3.2 人员信息数据字段

| 字段名       | 类型   | 必填 | 说明                           |
|--------------|--------|------|--------------------------------|
| 人员唯一标识 | 字符串 | 是   | 由外部系统提供，用于关联         |
| 姓名         | 字符串 | 是   | 由外部系统提供，用于展示与合同关联 |

#### 2.3.3 系统账号数据字段

| 字段名       | 类型   | 必填 | 说明                         |
|--------------|--------|------|------------------------------|
| 用户名       | 字符串 | 是   | 登录凭证，全局唯一             |
| 人员唯一标识 | 字符串 | 是   | 关联人员信息                   |
| 密码         | 字符串 | 是   | 加密存储（bcrypt / argon2）    |
| 状态         | 枚举   | 是   | 可选值：启用 / 停用            |

#### 2.3.4 业务规则

- 账号与人员为一对一关系：一个人员唯一标识仅允许绑定一个账号。
- 登录成功后，系统根据账号关联的人员唯一标识查询姓名，作为合同经办人的默认值。
- 账号停用后无法登录，已创建的历史合同不受影响。

### 2.4 定价管理

> **说明**：本模块管理供应商与物料之间的定价关系。定价录入即生效，不考虑折扣、阶梯价、有效期等复杂场景。

#### 2.4.1 功能列表

| 编号    | 功能名称     | 功能描述                                                         |
|---------|--------------|------------------------------------------------------------------|
| PRC-001 | 定价录入     | 为指定的供应商与物料组合录入单价                                   |
| PRC-002 | 定价编辑     | 修改已有定价的单价或备注等信息                                     |
| PRC-003 | 定价列表查询 | 支持按供应商、物料、品类、状态等条件查询定价记录                   |
| PRC-004 | 定价详情查看 | 查看定价的完整字段信息                                           |
| PRC-005 | 定价失效     | 将定价标记为"已失效"，失效后不可用于新合同的采购条目               |

#### 2.4.2 数据字段定义

| 字段名   | 类型   | 必填 | 说明                                         |
|----------|--------|------|----------------------------------------------|
| 定价编号 | 字符串 | 是   | 系统生成，遵循 `PRC-YYYY-NNNN` 编码规则        |
| 供应商   | 引用   | 是   | 关联供应商记录                               |
| 物料     | 引用   | 是   | 关联物料记录                                 |
| 单价     | 数值   | 是   | 单位：元（人民币），精度保留两位小数           |
| 币种     | 枚举   | 是   | 当前版本固定为人民币（CNY）                   |
| 状态     | 枚举   | 是   | 可选值：有效 / 已失效                          |
| 备注     | 文本   | 否   |                                              |

#### 2.4.3 业务规则

- **唯一有效定价**：同一"供应商 + 物料"组合在同一时刻至多存在一条"有效"状态的定价。
- **自动失效**：当为已有有效定价的"供应商 + 物料"组合新增定价时，系统自动将原有效定价的状态变更为"已失效"。
- **手动失效**：支持将有效定价手动置为"已失效"。
- **历史数据保护**：定价失效不影响已生效合同中的采购条目数据（采购条目为冗余存储）。
- **创建前置条件**：仅当供应商状态为"已准入"且物料状态为"启用"时，方可创建定价。
- **不维护变更历史**：定价编辑直接覆盖原有数据，不保留历史版本。

### 2.5 合同模板管理

合同模板按合同类型独立管理，每种合同类型可配置一个当前启用的模板。创建合同时，系统自动按合同类型匹配并绑定当前启用的模板，用于后续 PDF 渲染。

#### 2.5.1 功能列表

| 编号    | 功能名称         | 功能描述                                                         |
|---------|------------------|------------------------------------------------------------------|
| TPL-001 | 模板新增         | 新增合同模板，指定模板名称、关联的合同类型及 HTML 内容             |
| TPL-002 | 模板编辑         | 修改已有模板的名称、HTML 内容或变量定义；编辑后版本号自动递增       |
| TPL-003 | 模板版本查看     | 支持查看模板的当前版本号及历史版本信息（版本号，不保留历史完整内容） |
| TPL-004 | 模板启用/禁用    | 控制模板的启用状态；禁用后不可用于新合同的创建                     |
| TPL-005 | 模板变量定义     | 定义模板中可用的变量名称、类型与说明，用于渲染时的字段映射         |

#### 2.5.2 数据字段定义

| 字段名         | 类型   | 必填 | 说明                                                         |
|----------------|--------|------|--------------------------------------------------------------|
| 模板编码       | 字符串 | 是   | 系统生成，遵循 `TPL-YYYY-NNNN` 编码规则                        |
| 模板名称       | 字符串 | 是   | 如"NDA保密协议标准模板"                                       |
| 关联合同类型   | 枚举   | 是   | 可选值：NDA保密协议 / 多物品采购合同                           |
| HTML模板内容   | 文本   | 是   | 支持变量替换 `{{var}}`、数组循环 `{{#each items}}`、条件判断 `{{#if}}` 语法 |
| 变量定义       | JSON   | 是   | 描述模板中可用变量的名称、类型和说明，例如 `{"confidential_scope": {"type": "text", "label": "保密信息范围"}}` |
| 版本号         | 字符串 | 是   | 如 V1.0、V1.1；每次编辑后自动递增                             |
| 启用状态       | 枚举   | 是   | 可选值：启用 / 禁用                                            |
| 创建时间       | 时间戳 | 是   | 系统自动生成                                                   |
| 更新时间       | 时间戳 | 是   | 系统自动更新                                                   |

#### 2.5.3 业务规则

- **模板版本管理**：采用单表管理，版本号随编辑自动递增，不保留独立的历史版本实体。
- **合同绑定**：创建合同时自动绑定对应合同类型下"启用"状态且版本号最新的模板。
- **禁用约束**：模板禁用后，不影响已绑定该模板的历史合同；新创建的合同则不再绑定禁用模板。
- **变量定义作用**：变量定义用于约束合同中 `content` 字段的键名规范，并为前端录入界面提供字段提示。

### 2.6 合同管理

#### 2.6.1 概述

合同管理模块支持两种合同类型，设计上通过 `content` JSON 字段和模板机制实现内容可扩展：

| 合同类型       | 说明                               | PDF 导出特征                     |
|----------------|------------------------------------|----------------------------------|
| NDA保密协议    | 双方之间的保密约定，不涉及具体物品   | 纯文本段落形式，不含采购条目表格   |
| 多物品采购合同 | 包含多条采购明细的正式采购合同       | 含采购明细表格 + 段落式条款内容    |

后续如需新增合同类型，只需新增对应的合同模板并扩展 `content` 字段的键值定义，无需改动合同主表结构。

#### 2.6.2 核心数据模型

```
合同主表 (Contract)
  ├── content (JSONB)
  ├── 关联 → 供应商 (Supplier)
  ├── 关联 → 人员信息 (Person，冗余存储)
  ├── 关联 → 合同模板 (Template)
  └── 1 : N → 合同采购条目 (ContractEntry)
```

#### 2.6.3 功能列表

| 编号    | 功能名称       | 功能描述                                                             |
|---------|----------------|----------------------------------------------------------------------|
| CTT-001 | 合同创建       | 选择合同类型和供应商后创建合同，系统自动绑定对应类型的当前启用模板       |
| CTT-002 | 合同编辑       | 修改合同主信息、`content` 内容以及采购条目（仅草稿状态和退回草稿后）     |
| CTT-003 | 合同列表查询   | 支持按合同编号、名称、类型、供应商、状态、日期范围等条件检索             |
| CTT-004 | 合同详情查看   | 查看合同完整信息，包括主表字段、`content` 内容和采购条目                 |
| CTT-005 | 合同状态管理   | 控制合同生命周期状态流转：草稿 → 已生效 → 归档 / 作废                   |
| CTT-006 | PDF 导出       | 基于合同绑定的模板渲染合同数据并生成 PDF 文件                           |
| CTT-007 | 采购条目管理   | 多物品采购合同专属功能：添加、编辑、删除采购条目；支持从有效定价自动带出单价 |

#### 2.6.4 合同主表数据字段

| 字段名         | 类型   | 必填 | 说明                                                                   |
|----------------|--------|------|------------------------------------------------------------------------|
| 合同编号       | 字符串 | 是   | 系统生成，遵循 `CTT-YYYY-NNNN` 编码规则                                  |
| 合同名称       | 字符串 | 是   | 如"2026年度CPU采购合同"                                                 |
| 合同类型       | 枚举   | 是   | 可选值：NDA保密协议 / 多物品采购合同                                     |
| 供应商         | 引用   | 是   | 关联供应商记录                                                         |
| 经办人标识     | 字符串 | 是   | 关联人员唯一标识，创建时默认取当前登录用户                               |
| 经办人姓名     | 字符串 | 是   | 冗余存储，关联人员姓名                                                   |
| template_id    | 引用   | 是   | 关联合同模板，合同创建时自动绑定                                         |
| content        | JSONB  | 是   | 统一存储合同的个性化扩展字段（详见 2.6.5）                                |
| 合同总金额     | 数值   | 否   | 仅多物品采购合同：系统根据采购条目自动汇总，合同生效后锁定；NDA合同此字段留空 |
| 生效日期       | 日期   | 否   |                                                                        |
| 截止日期       | 日期   | 否   |                                                                        |
| 合同状态       | 枚举   | 是   | 可选值：草稿 / 已生效 / 归档 / 作废（流转规则见 2.6.7）                  |
| 我方主体名称   | 字符串 | 否   | 签署方企业名称，可在模板中引用                                           |
| 创建时间       | 时间戳 | 是   | 系统自动生成                                                           |
| 更新时间       | 时间戳 | 是   | 系统自动更新                                                           |

#### 2.6.5 content 字段说明

`content` 采用 JSONB 格式存储，其键名由对应合同模板的变量定义决定。两类合同的典型 `content` 结构示例如下：

**NDA保密协议 content 示例：**

```json
{
  "confidential_scope": "涉及甲方向乙方披露的全部技术资料、商业计划、客户信息等",
  "confidential_period": "自本协议签署之日起 3 年",
  "liability_overview": "违约方应赔偿守约方因此遭受的全部直接损失",
  "governing_law": "中华人民共和国法律，管辖法院为甲方所在地人民法院"
}
```

**多物品采购合同 content 示例：**

```json
{
  "delivery_location": "甲方指定仓库（四川省成都市高新区XX路XX号）",
  "payment_method": "货到验收合格后 30 日内支付全部款项",
  "acceptance_criteria": "按照甲方提供的技术规格书进行验收",
  "liability_clause": "逾期交货每日按合同总金额的 0.5% 支付违约金"
}
```

#### 2.6.6 合同采购条目数据字段

| 字段名   | 类型   | 必填 | 说明                                                           |
|----------|--------|------|----------------------------------------------------------------|
| 所属合同 | 引用   | 是   | 关联合同主表                                                   |
| 序号     | 整数   | 是   | 条目排序号，从 1 开始                                           |
| 物料     | 引用   | 是   | 关联物料记录                                                   |
| 物料名称 | 字符串 | 是   | 冗余存储，保障历史合同数据不受物料信息变更影响                   |
| 规格型号 | 字符串 | 否   | 冗余存储                                                       |
| 单价     | 数值   | 是   | 冗余存储，保障历史合同数据不受定价变更影响                       |
| 数量     | 数值   | 是   | 大于 0                                                         |
| 计量单位 | 字符串 | 是   | 冗余存储                                                       |
| 总价     | 数值   | 是   | 系统自动计算：`round(单价 × 数量, 2)`，不可手动编辑               |
| 备注     | 文本   | 否   |                                                                |

> **冗余策略**：物料名称、规格型号、单价、计量单位在采购条目中冗余存储，确保合同生效后即使原始物料信息或定价发生变化，已生效合同的采购明细不受影响，保证历史数据的完整性和可追溯性。

#### 2.6.7 合同状态流转

```
                     ┌─────────────────┐
                     │      草稿       │
                     │    (Draft)      │
                     └────────┬────────┘
                              │ 确认生效
                              ▼
                     ┌─────────────────┐
          ┌──────────│     已生效      │──────────┐
          │ 退回草稿  │   (Effective)   │  作废     │
          ▼          └────────┬────────┘          ▼
   ┌──────────┐              │            ┌──────────┐
   │  草稿    │              │ 归档        │  作废    │
   │ (Draft)  │              ▼            │ (Void)   │
   └──────────┘       ┌──────────┐       └──────────┘
                      │   归档    │
                      │(Archived)│
                      └──────────┘
```

**各状态说明：**

| 状态   | 可编辑 | 可导出PDF | 可执行操作                               |
|--------|--------|-----------|------------------------------------------|
| 草稿   | 是     | 是        | 确认生效（→ 已生效）                      |
| 已生效 | 否     | 是        | 退回草稿（→ 草稿，可重新编辑）、作废、归档 |
| 作废   | 否     | 否        | 仅可查看，不可恢复                         |
| 归档   | 否     | 是        | 仅可查看，不可退回                         |

**流转约束：**

- 草稿确认生效后，合同主表、`content` 及采购条目全部锁定，不可编辑。
- 已生效合同退回草稿后，编辑内容再次确认生效方可使合同重新生效。
- 作废为不可逆操作，作废后合同仅保留历史查询，不参与正常业务列表统计。
- 归档后不可退回草稿，仅用于完结合同的历史存档。

#### 2.6.8 创建前置条件

- 供应商状态必须为"已准入"。
- 对应的合同类型必须存在至少一个启用状态的模板。
- 多物品采购合同至少包含一条采购条目方可确认生效。

### 2.7 PDF 导出

#### 2.7.1 通用机制

- 基于服务器端 HTML 模板引擎将合同数据渲染为 HTML，再转换为 PDF。
- 渲染时将合同主表字段、`content` JSON 对象和采购条目数组整体传入模板引擎。
- PDF 支持在线预览和文件下载两种方式，文件命名格式为 `合同编号-合同名称.pdf`。
- 企业 Logo、中文字体（宋体/黑体）、水印样式在模板的 HTML 中配置。

#### 2.7.2 NDA保密协议 PDF

- 以**纯文本段落**形式呈现，不包含任何表格。
- 模板通过 `{{content.字段键}}` 语法将 `content` 中各字段渲染为段落标题及其内容。
- PDF 包含：企业 Logo / 公司名称、协议标题、合同编号、甲乙双方信息、保密条款段落、签署区域。

#### 2.7.3 多物品采购合同 PDF

- 采购物品明细以**HTML 表格**形式呈现，列包含：序号、物品名称、规格型号、单价、数量、计量单位、总价。
- 表格末尾行显示**合计总金额**（`合同总金额` 字段值）。
- `content` 中的扩展字段以段落形式呈现在采购明细表格之后。
- PDF 包含：企业 Logo / 公司名称、合同标题、合同编号、甲乙双方信息、采购明细表格、扩展条款段落、签署区域。

## 3. 非功能需求

### 3.1 性能指标

| 指标项               | 要求                                                   |
|----------------------|--------------------------------------------------------|
| 列表查询响应时间     | 单表列表（≤ 2000 条记录）≤ 2 秒                         |
| 详情查询响应时间     | ≤ 1 秒                                                 |
| PDF 生成时间         | 含 ≤ 10 条采购条目的合同 PDF 生成 ≤ 5 秒                |
| PDF 生成时间（上限） | 含 ≤ 50 条采购条目的合同 PDF 生成 ≤ 15 秒               |
| 并发支持             | 系统应支持至少 10 个用户同时进行查询操作而不出现明显延迟 |

### 3.2 安全要求

| 要求项             | 具体描述                                                             |
|--------------------|----------------------------------------------------------------------|
| 密码存储           | 使用 bcrypt 或 argon2 等单向哈希算法加密存储，禁止明文或可逆加密       |
| 接口鉴权           | 除登录接口外，所有业务 API 需验证用户认证状态（JWT Token 或 Session）  |
| 输入校验           | 所有用户输入需在服务端进行二次校验，防止 SQL 注入、XSS 等攻击         |
| 敏感信息           | 日志中禁止输出密码、Token 等敏感数据                                  |
| HTTPS              | 生产环境必须启用 HTTPS 传输                                           |

### 3.3 兼容性范围

| 维度       | 要求                                             |
|------------|--------------------------------------------------|
| 浏览器     | Chrome 100+、Edge 100+、Firefox 100+ 最新两个主版本 |
| 操作系统   | 服务端支持 Linux（推荐 Ubuntu 22.04+）部署           |
| 屏幕分辨率 | 前端适配桌面端 1366×768 及以上分辨率                 |
| PDF 阅读器 | 导出的 PDF 兼容 Adobe Acrobat Reader 及主流浏览器内置 PDF 查看器 |

### 3.4 可用性要求

| 要求项             | 具体描述                                                           |
|--------------------|--------------------------------------------------------------------|
| 操作确认           | 不可逆操作（合同作废、定价失效、供应商淘汰等）执行前弹出二次确认对话框 |
| 错误提示           | 操作失败时提供明确的错误原因说明，而非仅显示技术错误码               |
| 表单校验           | 前端实时校验 + 提交时校验，必填字段遗漏时明确高亮提示                 |
| 空状态             | 列表无数据时展示引导性提示文字（如"暂无供应商记录，点击新增"）       |

### 3.5 数据一致性与完整性

| 要求项             | 具体描述                                                               |
|--------------------|------------------------------------------------------------------------|
| 引用完整性         | 删除被其他实体引用的记录时，系统必须返回明确的冲突提示，禁止级联删除     |
| 合同数据锁定       | 合同生效后锁定主表、`content` 及采购条目，禁止任何编辑操作               |
| 冗余字段一致性     | 采购条目中的冗余字段（物料名称、规格型号、单价等）在合同创建时从源头拷贝，生效后不再与源数据同步 |
| 事务性             | 涉及多表写操作（如创建合同同时写入采购条目）须在同一数据库事务中完成     |

### 3.6 可维护性与可扩展性

| 要求项             | 具体描述                                                             |
|--------------------|----------------------------------------------------------------------|
| 合同类型扩展       | 新增合同类型仅需：新增合同模板 + 定义 `content` 键值规范，无需修改合同主表结构 |
| 编码规则扩展       | 新增实体类型时遵循统一的 `前缀-年份-流水号` 编码框架                   |
| 日志记录           | 关键业务操作（合同状态变更、定价变更等）需记录操作时间和操作人           |
| 配置外部化         | 企业名称、Logo URL 等环境相关配置项应通过配置文件管理，避免硬编码       |

## 4. 数据模型总览

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│  供应商   │     │  品类    │     │ 合同模板  │
│ Supplier │     │ Category │     │ Template │
└────┬─────┘     └────┬─────┘     └────┬─────┘
     │ 1:N            │ 1:N            │ 1:N
     ▼                ▼                ▼
┌──────────┐     ┌──────────┐     ┌──────────┐
│   定价   │     │   物料   │     │ 合同主表  │
│  Price   │     │ Material │     │ Contract  │
└──────────┘     └────┬─────┘     └────┬───┬──┘
                       │               │   │
                       │ 1:N           │   │ N:1
                       ▼               │   ▼
                 ┌──────────┐          │ ┌──────────┐
                 │ 合同条目 │ ◄────────┘ │ 人员信息  │
                 │  Entry   │            │  Person   │
                 └──────────┘            └──────────┘
```

**关系说明：**

| 关系               | 说明                                                                 |
|--------------------|----------------------------------------------------------------------|
| 供应商 → 定价      | 一个供应商可有多条定价（关联不同物料）                                 |
| 供应商 → 合同主表  | 一个供应商可关联多个合同                                               |
| 品类 → 物料        | 一个品类下包含多种物料；物料必须归属一个品类                           |
| 物料 → 定价        | 一个物料可与多个供应商形成定价；同一"供应商+物料"至多一条有效定价       |
| 合同模板 → 合同主表 | 一个模板可被多个合同引用；一个合同绑定一个模板                          |
| 合同主表 → 合同条目 | 一个合同包含多条采购条目（仅多物品采购合同）；条目从属于合同             |
| 人员信息 → 合同主表 | 一个人员可经办多个合同                                                 |

## 5. 典型业务流程

### 5.1 主流程：供应商准入 → 定价 → 采购合同

```
步骤1  [采购员] 录入供应商基础信息
       → 供应商状态 = 已准入

步骤2  [采购员] 为供应商 × 物料组合录入定价
       → 前置条件：供应商为"已准入"、物料为"启用"
       → 定价状态 = 有效

步骤3  [采购员] 创建多物品采购合同
       → 选择供应商，系统自动绑定当前启用的采购合同模板
       → 添加采购条目：
           - 选择物料后，系统自动带出该供应商对应物料的当前有效定价（单价、物料名称、规格、单位）
           - 允许手动覆盖带出的单价
           - 填写数量，系统自动计算总价
       → 填写 content 扩展字段（交货地点、付款方式、验收标准等）
       → 合同状态 = 草稿

步骤4  [采购员] 确认合同内容，标记为已生效
       → 系统校验：采购条目数量 ≥ 1
       → 系统自动计算并锁定"合同总金额"
       → 锁定合同主表、content 及采购条目数据

步骤5  [采购员] 导出合同 PDF
       → 系统基于绑定模板渲染合同数据生成 PDF

步骤6  [采购员] 归档合同（可选）
       → 合同状态 = 归档
```

### 5.2 NDA保密协议 完整流程

```
步骤1  [采购员] 创建 NDA 保密协议合同
       → 选择供应商
       → 系统自动绑定当前启用的 NDA 模板
       → 填写 content 扩展字段（保密范围、保密期限、违约责任等）
       → 合同状态 = 草稿

步骤2  [采购员] 确认合同内容，标记为已生效
       → 锁定合同主表及 content 数据

步骤3  [采购员] 导出 NDA 协议 PDF

步骤4  [采购员] 归档合同（可选）
```

### 5.3 合同退回修改流程

```
步骤1  [采购员] 将已生效合同退回草稿
       → 合同状态: 已生效 → 草稿
       → 解除数据锁定，允许编辑

步骤2  [采购员] 修改合同内容（主表字段、content 或采购条目）

步骤3  [采购员] 重新确认生效
       → 合同状态: 草稿 → 已生效
       → 重新锁定数据
```

### 5.4 模板维护流程

```
步骤1  [采购员] 新增或编辑合同模板
       → 指定模板名称、关联合同类型
       → 编写 HTML 模板内容，定义变量
       → 启用模板（同一类型仅最新版本处于启用状态）

步骤2  [采购员] （可选）禁用旧模板
       → 禁用后不影响已绑定该模板的历史合同

步骤3  创建新合同时，系统自动绑定当前启用的最新版本模板
```
